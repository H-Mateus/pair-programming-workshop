# Pair Programming: Participant Reference

## The two roles

**Driver** — controls the keyboard and types. Focuses on the immediate task: writing, reading, and editing code.

**Navigator** — watches, thinks ahead, and asks questions. Focuses on the bigger picture: does this approach make sense, are there edge cases, what should happen next?

Switch roles regularly — every 10–15 minutes, or when you reach a natural pause.

## Before you start

- Give your partner a 2-minute tour of your code: what does it do, what are you working on, where are you stuck?
- Agree who drives first

## While you're pairing

**If you're the driver**
- Talk through what you're doing as you type — "I'm looping over the samples here because..."
- Ask the navigator before making big changes — "should I refactor this function or keep going?"

**If you're the navigator**
- Ask questions freely — "what does this variable hold?" or "why does this need to be a list?"
- Don't grab the keyboard; instead say what you'd try and let the driver type it
- Watch for typos and logic errors the driver might miss

**Both of you**
- It's fine not to solve the problem — the conversation is the point
- Messy, unfinished code is normal and expected; no judgement
- If you're stuck, say so — "I have no idea what this does" is a useful thing to say

## What to bring

A real piece of code you're working on. It doesn't need to:
- Be finished
- Work correctly
- Be well-organised
- Be in a language your partner knows

If you're not sure what to bring, pick a script you run regularly. There's almost always something to improve, explain, or make clearer.

## Things that often come up in sessions

- A tool or library your partner uses that you haven't seen before
- A different way of approaching a problem you've been stuck on
- Something you've been doing for years that turns out to have a neater solution
- A bug that you spotted the moment you had to explain the code out loud
