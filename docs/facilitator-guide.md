# Facilitator Guide

## Before the event

**Logistics**
- Book a room with tables big enough for two people to sit side by side at one laptop
- Ask participants to bring their laptops with a piece of code they're actively working on — doesn't need to be finished or even working
- Aim for 6–16 participants; pairs of 2 work best, so keep numbers even if possible
- Prepare a random pairing list in advance (or use a random name picker on the day)

**What participants should bring**
- A laptop
- A piece of code they'd like to improve, debug, or understand better
- No special preparation beyond that — the whole point is to work on real problems

## The intro talk (~20 minutes)

This sets up the mental model participants need. Suggested structure:

1. **Why we're here** (5 min) — researchers who code often work in isolation; pair programming is a way to learn from peers without needing formal training
2. **How pair programming works** (5 min) — explain driver/navigator roles; emphasise that switching regularly is the key mechanic
3. **What makes a good session** (5 min) — talk out loud, ask questions freely, no judgement on messy code
4. **What to expect today** (5 min) — walk through the format, answer questions

Key points to land:
- The navigator's job is to ask questions and think ahead, not to watch silently
- "I don't understand what this does" is a useful thing to say, not an embarrassing one
- It's fine if you don't solve anything in 25 minutes — the conversation is the value

## Running the rounds

**Pairing**
- Reveal pairings at the start of each round
- Mix experience levels where possible — a more experienced coder paired with a less experienced one usually works well in both directions
- In round 2, try to pair people who haven't worked together yet

**Starting a round**
- Remind pairs to decide who drives first
- Suggest the owner of the code gives a 2-minute tour of what it does and what they're stuck on
- Set a visible timer

**Mid-round**
- Walk the room quietly; you don't need to intervene unless a pair looks stuck or silent
- If a pair finishes their problem early, prompt them to switch driver/navigator and keep going, or start on the second person's code early

**Ending a round**
- Give a 2-minute warning before the timer
- Ask pairs to reach a stopping point rather than stopping mid-thought

## Debrief (~15 minutes)

Bring the group back together and run a brief retrospective. Useful prompts:
- What did you learn that surprised you?
- Did you see a tool or technique you hadn't heard of?
- What would make the next session more useful?

Collect this feedback — it's useful for iterating the format and for SSI reporting.

## Common issues

**"I don't have any code to bring"** — suggest they bring a script they use regularly, even if it works fine; there's usually something to improve or explain

**Very mismatched skill levels** — usually fine; the less experienced person asks good questions, the more experienced person learns by explaining

**Pairs going quiet** — sometimes people need a prompt; suggest the navigator ask "walk me through what this line does"

**Code that won't run on the navigator's machine** — that's fine; the driver controls the keyboard, the navigator looks at the screen
